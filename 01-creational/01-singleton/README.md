## 🎬 场景：办公室只有一台打印机

想象你在一家 30 人的小公司，办公区角落里放着一台打印机。整个公司的人共用它：

- 📄 小明打印月度报告
- 📄 小红打印合同
- 📄 老板打印 PPT

这台打印机有一个**共享的打印队列** —— 谁先点"打印"，谁的任务先出纸。它还能统计"今天全司一共打了多少张"，用来月底算耗材预算。

**关键问题**：如果公司给**每个人**都发一台打印机，会发生什么？

1. 💸 **太贵了** —— 一台打印机几千块，30 个人就是十几万
2. 🤷 **统计乱套** —— 每台打印机自己算自己的，算全司用量要一个个去查
3. 😵 **任务不同步** —— 小明不知道小红的任务排在哪，打印室里变成乱麻

所以现实中，办公室**只有一台**打印机，所有人**共享**它。这就是单例模式的直觉。

## 💡 第二步：单例模式登场

### 定义

> **单例模式（Singleton）**：确保一个类**只有一个实例**，并提供一个**全局访问点**。
> —— GoF《设计模式》（1994）

翻译成人话：**我这个类，你甭想 new 第二个。想用？只能通过我指定的方法拿**。

### 核心结构

```
┌──────────────────────────┐
│      Singleton           │
├──────────────────────────┤
│ - instance : Singleton   │  ← 类内部持有唯一实例
├──────────────────────────┤
│ - Singleton()            │  ← 构造器私有，外部不能 new
│ + getInstance()          │  ← 全局访问点，返回唯一实例
└──────────────────────────┘
```

**三个要点**，缺一不可：

1. **构造器 `private`** —— 把 `new` 的入口堵死
2. **类内持有一个静态实例** —— `private static` 字段
3. **暴露 `getInstance()` 静态方法** —— 唯一获取入口

## 🛠 第三步：五种实现走一遍

打开 [code/SingletonDemo.java](code/SingletonDemo.java)，里面用**同一个"打印机"场景**实现了五种单例，注释详细。先跑一下：

```bash
java SingletonDemo.java
```

本节我们**重点讲三种**（🥇🥈🥉），另外两种（懒汉不安全 / Holder）当扩展阅读。

### 🥇 饿汉式（Eager）—— 起步推荐

```java
class EagerPrinter {
    private static final EagerPrinter INSTANCE = new EagerPrinter();  // 类加载就创建
    private EagerPrinter() {}                                          // 禁止外部 new
    public static EagerPrinter getInstance() { return INSTANCE; }      // 唯一入口
}
```

**优点**：线程安全（JVM 类加载机制保证）、代码极简
**缺点**：类加载就创建，哪怕没人用也占内存
**什么时候用**：**大多数场景**。对绝大多数应用，那点内存完全不是问题

### 🥈 双检锁 DCL（Double-Checked Locking）—— 高并发老将

```java
class DCLPrinter {
    private static volatile DCLPrinter instance;   // volatile 必须加！
    private DCLPrinter() {}
    public static DCLPrinter getInstance() {
        if (instance == null) {                    // 第一次检查：无锁，快
            synchronized (DCLPrinter.class) {
                if (instance == null) {            // 第二次检查：加锁后再确认
                    instance = new DCLPrinter();
                }
            }
        }
        return instance;
    }
}
```

**关键**：`volatile` **不能省**。原因：`new` 操作实际上是"分配内存 → 调构造器 → 赋值给变量"三步，JVM 可能重排成"分配内存 → 赋值 → 调构造器"。另一个线程拿到还没初始化完的对象，炸了。`volatile` 禁止重排。

**什么时候用**：懒加载 + 高并发 + 实例创建成本极高（比如加载配置文件、连数据库）

### 🥉 枚举（Enum）—— 最强之选

```java
enum EnumPrinter {
    INSTANCE;                         // 👈 就这一行，整个 JVM 唯一
    public void print(String doc) { /* ... */ }
}

// 使用：
EnumPrinter.INSTANCE.print("报告.pdf");
```

**Joshua Bloch 在《Effective Java》里说："单特权元素的枚举类型，是实现 Singleton 的最佳方法。"**

**为什么最强**：

- ✅ 线程安全（JVM 保证）
- ✅ **防反射攻击**（用反射破坏单例是一招，但反射不能 new 枚举）
- ✅ **防序列化破坏**（反序列化默认会生成新对象，但枚举不会）
- ✅ 代码最短

**唯一的"缺点"**：不能继承其他类，有些人觉得"枚举用来做单例怪怪的"。但那是习惯问题。

### 🔹 另外两种（知道即可）

- **懒汉式（线程不安全）**：典型的反面教材，多线程下会创建多个实例。只在单线程或教学场景出现
- **静态内部类（Holder Idiom）**：利用 JVM 类加载机制实现懒加载 + 线程安全，优雅但理解成本略高。看 `SingletonDemo.java` 里有实现

## 🧭 决策小抄

```
我需要单例 → 是不是"创建成本低、启动就要用"的？
              ├── 是 → 用饿汉式（🥇 起步首选）
              └── 否 → 是不是"初始化很重、可能不会用到"？
                       ├── 是 → 用 DCL 或 静态内部类
                       └── 不确定 → 用枚举（🥉 最安全）

我需要"绝对不被反射/序列化破坏"？
              → 直接上枚举
```

## 🌍 真实世界里的单例

你每天用的这些都是单例：

| 在哪里 | 谁是单例 | 用途 |
|--------|---------|------|
| JDK | `Runtime.getRuntime()` | JVM 运行时环境 |
| JDK | `Desktop.getDesktop()` | 桌面交互 |
| Spring | 默认所有 `@Bean` / `@Component` | IoC 容器里的 bean 默认是 singleton |
| Java 日志 | Log4j 的 `LoggerContext` | 全局日志上下文 |
| Android | `Application` 类 | 整个 App 唯一 |

打开 JDK 源码搜 `private static final ... INSTANCE = ` 你会被吓到。

## ⚠️ 警告：单例的"黑历史"

单例是 GoF 23 种模式里**被骂得最惨**的一个。很多人说它是"反模式"。原因：

### 1. 它是"披着模式外衣的全局变量"
全局变量的所有坏处，单例一个不落：任何地方都能改它、任何地方都依赖它、测试时无法隔离。

### 2. 它让单元测试变得超痛苦
```java
public void doBusiness() {
    ConfigManager.getInstance().getConfig(...);  // 这行代码怎么 mock？
}
```
类内部直接调 `getInstance()`，在测试里你**没办法注入假数据**。

### 3. 它隐藏了依赖关系
看类的构造器和方法签名，你看不出它依赖什么。等上线出 bug 才发现它偷偷用了某个全局状态。

### 👉 现代做法：用"依赖注入"代替单例
Spring 里的 `@Autowired` 本质上是"单例模式 + 依赖注入"：对象依然是全局唯一的，但**通过参数传进来**，不是类里面自己 `getInstance()`。这样可测、可替换、依赖显式。

> 💡 这条我们到 [第六阶段 · 依赖注入](../../06-enterprise/02-dependency-injection/) 会详细讲。

### 🚫 什么时候**千万别**用单例
- 你只是为了"方便访问" → 用依赖注入
- 你的对象**可能需要多个实例**（哪怕只是测试） → 别用
- 你的对象**持有可变状态**且被多线程使用 → 小心，可能需要其他方案
