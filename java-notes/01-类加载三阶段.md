# 类加载三阶段 · Loading → Linking → Initialization

> 📍 来源：设计模式学习 **单例 V1 饿汉式 / V4 静态内部类**
> 🎯 回答的核心问题："**类里面怎么可以 new 自己？**"

---

## 🤔 场景问题

写单例 V1 饿汉式时，你看到这行代码：

```java
class EagerPrinter {
    private static final EagerPrinter INSTANCE = new EagerPrinter();
    //                                             ↑
    //                              类在自己内部 new 自己？不是鸡生蛋悖论吗？
    private EagerPrinter() {}
}
```

困惑：**"类还没造好怎么就 new 自己？"**

答案藏在 JVM 的类加载机制里。

---

## 🧠 核心结论

> Java 类从"字节码文件"变成"能被使用的状态"，要经过**三个阶段**。
> **构造器在第 1 阶段就已经"存在"，但直到第 3 阶段才被"执行"**。

所以"`static INSTANCE = new 自己`" 不是悖论 —— **构造器早就存在了**（第 1 阶段），第 3 阶段只是调它而已。

---

## 📊 三个阶段详解

### 阶段 ① Loading（加载）
JVM 读 `EagerPrinter.class` 字节码进内存。

**这一步发生了什么**：
- 把字节码放入方法区
- **类的"结构"定义就位**：字段名、方法名、构造器的字节码都在了
- **但还没有任何对象**，连字段都还没分配内存

### 阶段 ② Linking（链接）
三个子步骤：

| 子阶段 | 做什么 |
|--------|--------|
| Verification 验证 | 检查字节码合法（防恶意字节码） |
| **Preparation 准备** | **给 static 字段分配内存 + 赋默认值** |
| Resolution 解析 | 符号引用 → 直接引用 |

**关键在 Preparation**：

```java
static Object obj;       // obj 被分配内存，默认值 null
static int count;        // count 被分配内存，默认值 0
static boolean ok;       // ok  被分配内存，默认值 false

// 对于 V1 单例：
private static final EagerPrinter INSTANCE = new EagerPrinter();
//                                            ↑
//                   这一步 INSTANCE 默认值是 null，还没执行右边的 new
```

**此时 INSTANCE = null**，还没有真正赋值。

### 阶段 ③ Initialization（初始化）
执行 `<clinit>` —— 编译器收集所有 `static` 赋值语句和 `static {}` 块生成的方法。

```java
// 编译器生成的 <clinit> 方法，大致是：
<clinit>() {
    INSTANCE = new EagerPrinter();   // 👈 这一步终于执行 new
}
```

**到此**：
- `new EagerPrinter()` 被调用
- 构造器执行，对象创建
- INSTANCE 指向这个对象

---

## 🎬 完整时间轴

```
JVM 第一次遇到 EagerPrinter
       ↓
  ┌──────────────────┐
  │ ① Loading        │  读字节码 → 类结构就位，构造器已定义
  └──────────────────┘
       ↓
  ┌──────────────────┐
  │ ② Linking        │  INSTANCE 分配内存，值 = null
  └──────────────────┘
       ↓
  ┌──────────────────┐
  │ ③ Initialization │  执行 INSTANCE = new EagerPrinter()
  │                  │  调构造器 → 对象就绪 → INSTANCE 指向它
  └──────────────────┘
       ↓
  以后访问 INSTANCE → 直接返回，不会再走这三步
```

---

## 💡 为什么"类里 new 自己"不矛盾

| 时刻 | 构造器"定义" | 构造器"执行" | INSTANCE 值 |
|------|------------|------------|-------------|
| Loading 后 | ✅ 存在 | ❌ 未执行 | 还没内存 |
| Linking 后 | ✅ 存在 | ❌ 未执行 | null |
| Initialization 中 | ✅ 存在 | ⏳ 正在执行 | null（指向将被 new 的对象） |
| Initialization 后 | ✅ 存在 | ✅ 已完成 | 指向对象 |

**关键**：构造器"定义存在"和"被执行"是两回事。第 ① 步让定义就位，第 ③ 步才执行 —— **所以不是鸡生蛋悖论**。

---

## 🧭 触发类加载的时机

JVM **不会**一启动就加载所有类。类加载是**按需触发**的。典型触发场景：

1. `new Foo()` —— 创建实例
2. `Foo.someStaticMethod()` —— 调静态方法
3. `Foo.someStaticField` —— 读/写静态字段（但 `final` 常量除外，会被编译器内联）
4. `Class.forName("Foo")` —— 反射加载
5. 子类触发 → 父类先加载

**V4 静态内部类单例的妙用**：利用"内部类不会随外部类加载"这条规则，让 `Holder` 推迟到第一次 `getInstance()` 才加载，实现懒加载。

---

## ⚠️ 常见误区

### 误区 1："类加载就是 new 对象"
**不是**。类加载只是让 JVM 认识这个"类型"，**不会自动创建实例**（除了 static 字段上写的那个）。

### 误区 2："static 字段在声明时就有值"
**不是**。Preparation 阶段先赋默认值（null / 0 / false），Initialization 阶段才赋真正的值。这两步是**分开**的。

### 误区 3："构造器要等类加载完才'有'"
**反了**。构造器在 Loading 后就已经"存在"（只是还没被调用）。Initialization 阶段调用它，不会有问题。

## 🎁 一个有趣的细节

你可以写一个"空类"，启动时打印类加载信息：

```java
class Demo {
    static {
        System.out.println("Demo 类被初始化了！");   // <clinit> 里的代码
    }
}

public class Main {
    public static void main(String[] args) {
        System.out.println("main 开始");
        Demo d = new Demo();  // 👈 这里才触发 Demo 的类加载 + 初始化
    }
}
```

输出：
```
main 开始
Demo 类被初始化了！
```

**说明类加载是"碰到了才加载"，不是启动就全加载**。

## 📌 一句话总结

> **类加载 = 三步走：字节码进内存 → 字段分配默认值 → 执行 static 赋值。构造器在第一步就已存在，第三步被调用时绝对合法。这就是单例饿汉式 / 静态内部类能正常工作的底层原理。**
